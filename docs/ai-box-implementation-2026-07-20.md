# AI BOX — 구현 절차

기획: [`ai-box-plan-2026-07-20.md`](ai-box-plan-2026-07-20.md)

깃허브 MD 연동은 서버 없이 정적 앱만으로 가능하므로 1단계로 먼저 간다.
구글드라이브는 인증·CORS 제약이 있어 2단계로 분리한다.

---

## 1단계 — 깃허브 MD → 웹앱 자동 갱신

### 1-1. 글 저장소 구조 확정

레포에 카테고리당 MD 파일 하나. 파일명 = 카테고리.

```
posts/자유.md
posts/정보.md
posts/그림프롬프트.md
posts/프롬프트.md
```

### 1-2. 글 형식(포맷) 확정

AI가 쓰기 쉽고 앱이 파싱하기 쉬운 최소 규칙:

```markdown
## 글 제목 (2026-07-20)

본문. 여러 줄 가능, 코드 블록 가능.
```

- `## ` 헤딩 하나가 카드 하나
- 새 글은 파일 **맨 위**에 추가 (최신순)
- 헤딩 끝의 `(YYYY-MM-DD)`는 날짜 표기 — 없으면 날짜 없이 표시

### 1-3. 웹앱에서 읽기

**⚠️ 정정 — 이 레포는 비공개(private)라 raw 익명 접근이 안 된다.**
`raw.githubusercontent.com`은 공개 레포만 익명 fetch가 되고, 비공개 레포는
모든 파일이 404다(기존 계획의 "raw는 CORS 허용" 전제는 공개 레포 한정).
그래서 **GitHub API + 토큰** 방식으로 바꿨다:

- `https://api.github.com/repos/LUNA-X59Q/LUNA-WORLD/contents/posts/<카테고리>.md?ref=main`
  를 `fetch`, 헤더 `Accept: application/vnd.github.raw`(본문 그대로 반환) +
  `Authorization: Bearer <토큰>`. api.github.com은 CORS(인증 포함)를 허용한다.
- **토큰은 코드에 넣지 않는다.** 앱 "설정(⚙)"에서 사용자가 붙여넣어
  그 기기의 `localStorage`(`ai-box-gh-token`)에만 저장 → 레포·타인에게 노출 안 됨.
  토큰이 없거나 무효면 404 → 잠금 안내 문구 표시. 공개 레포면 토큰 없이도 됨.
- 파일이 없으면(404) 그 카테고리는 깃허브 글 0개로 처리
- 받은 MD를 `## ` 기준으로 잘라 카드로 렌더 (본문은 최소 마크다운:
  줄바꿈·코드블록·굵게 정도만, 외부 라이브러리 없이)
- 성공한 응답은 localStorage에 캐시 → 오프라인·실패 시 마지막 판 표시

**필요한 토큰**: GitHub → Settings → Developer settings →
Fine-grained tokens → 이 레포만 선택 → 권한 **Contents: Read-only**.
(최소 권한 · 만료일 지정 · 언제든 폐기 가능. 노출돼도 이 레포 읽기 전용뿐.)

주의: API 응답 캐시 회피용 타임스탬프를 붙이고, **새로고침 버튼**을 두고
앱 열 때마다 자동 fetch한다.

### 1-4. UI 통합

- 깃허브 글 카드에 출처 배지(예: `GitHub`) — 로컬 카드와 구분
- 깃허브 카드는 앱에서 삭제 불가(원본이 레포이므로), 로컬 카드만 삭제 가능
- 정렬: 깃허브 글(파일 순서 = 최신순) 먼저, 그 아래 로컬 카드 — 또는
  날짜 있으면 섞어서 최신순

### 1-5. 검증

- posts/ 파일에 샘플 글 1개 넣고 → 앱에서 카드로 보이는지
- AI(클로드 등)에게 "그림프롬프트에 글 하나 올려줘" 시켜서
  커밋 → 푸시 → 앱 새로고침 → 카드 등장까지 전체 흐름 확인
- 파일 없음(404)·형식 어긋난 글·오프라인 3가지 경우 확인

**1단계 끝나면 기획의 핵심("AI가 MD에 쓰면 앱에 뜬다")은 완성.**

## 2단계 — 구글드라이브 MD 읽기 (선택)

깃허브보다 제약이 많아 뒤로 뺀다. 두 가지 경로 중 택1:

- **A. Drive API + API 키**: 파일을 "링크가 있는 모든 사용자" 공개로 두고
  `www.googleapis.com/drive/v3/files/<ID>?alt=media&key=<API키>` fetch.
  googleapis.com은 CORS 허용이라 클라이언트에서 됨.
  단점: API 키가 HTML에 노출됨(읽기 전용 공개 파일이므로 위험은 낮음),
  구글 클라우드 콘솔에서 키 발급 필요
- **B. 경유 서버**: Cloud Run에 작은 프록시를 두고 서버가 Drive를 읽음.
  키가 숨겨지지만 "서버 0 유지" 운영 수칙과 충돌 → 웬만하면 A

### 구현 상태 (A안 — 폴더 지정 방식, 키는 설정에 저장)

**파일을 하나하나 등록하지 않고, 카테고리별 "폴더"를 지정하면 그 폴더 안의
모든 문서를 조회한다.** `index.html`의 드라이브 소스는:

- `files.list?q='<폴더ID>' in parents and trashed=false&orderBy=modifiedTime desc`
  로 폴더 안 문서 목록(최신순)을 받고, 하위 폴더는 제외한다.
- 각 문서를 읽는다: 일반 파일은 `?alt=media`, **구글 문서**(Docs)는
  `/export?mimeType=text/plain`. 둘 다 googleapis라 CORS로 브라우저에서 됨.
- 각 문서를 파싱: `## 제목 (날짜)` 헤딩이 있으면 그 단위로 카드, 없으면
  **문서 하나를 카드 하나로**(제목=파일명, 날짜=수정일 또는 파일명의 `(날짜)`).
- 배지는 초록색 `Drive`. 파싱 결과(글 배열)를 `localStorage`에 캐시
  (`ai-box-drive-posts`) → 오프라인·실패 시 마지막 판.

**키를 코드에 하드코딩하지 않는다** — 깃허브 토큰과 동일하게 앱 설정(⚙)에서
받아 그 기기의 `localStorage`에만 저장(`ai-box-drive-key`,
`ai-box-drive-folders`). 설정 화면은 `GitHub` / `구글 드라이브` 두 구획.

**사용자가 준비할 것** (앱 ⚙ 설정에 입력, 채팅·레포에는 넣지 않음)

1. **API 키** — 구글 클라우드 콘솔에서 프로젝트 생성 →
   "API 및 서비스 → 라이브러리"에서 **Google Drive API** 사용 설정 →
   "사용자 인증 정보 → API 키". 노출 최소화를 위해 키 제한에서 API를
   **Drive API로 제한**하고, 가능하면 HTTP 리퍼러(앱 도메인)도 제한.
2. **폴더 ID** — 카테고리별 폴더를 만들어 "링크가 있는 모든 사용자(뷰어)"로
   공개 → 폴더 링크 `https://drive.google.com/drive/folders/<ID>`의 `<ID>`를
   설정의 해당 칸에 입력. 이후 그 폴더에 문서를 넣기만 하면 앱에 뜬다.
   글 형식은 깃허브와 동일(`## 제목 (YYYY-MM-DD)`)이거나, 한 문서=한 카드.

API 키는 링크 공개된 읽기 전용 폴더/문서에만 접근하므로 노출 위험이 낮다.

**B. 경유 서버**(Cloud Run 프록시로 키 은닉)는 "서버 0 유지" 수칙과 충돌해
채택하지 않았다.
